## Challenges & Growth Areas

### Story Context Spec Inaccuracies

5 inaccuracies across 5 stories:
- 8.2: `assert_cmd` v2 has no `cargo` feature (spec said to add it)
- 8.2: `QueueName` has no `AsRef<str>` (spec assumed it did)
- 8.4: `TaskError::handler()` doesn't exist (spec referenced it)
- 8.5: `start_workers(queue, concurrency)` doesn't exist (spec referenced it)
- 8.5: `docker/k8s/` directory doesn't exist (spec referenced it)

**Root cause:** Context-creation agent relied on architecture docs rather than verifying against the actual codebase. The implementation agent corrected all 5 on the fly, but less experienced developers following specs literally would hit walls.

### E2E Test Race Conditions

Story 8.2's CLI-to-REST consistency test submitted to the worker queue, and the worker changed task status between CLI output and REST read. Fixed by submitting to a queue without workers, but discovered through test failure rather than by design.

### OTel-Prometheus Naming Quirks

Story 8.3's metrics test broke because OTel-Prometheus doubles the `_seconds` suffix and adds `_total` to counters. Fixed with `contains()` and regex matching. This is a known quirk that should be a documented project-level pattern.

### Review Patches Concentrated in Examples

Story 8.4 had 4 out of 9 total review patches — deadlock risk, migration race, polling loop timeout, unguarded retrieval. Examples are user-facing code that new developers copy first; bugs there erode trust faster than bugs in internals.

## Key Lessons

1. **Story context spec accuracy matters** — 5 inaccuracies across 5 stories is a pattern. Context-creation agents should verify method signatures, feature flags, and file paths against the actual codebase.

2. **E2E tests need queue isolation by design** — consistency tests across interfaces must submit to queues without active workers. This should be a documented pattern.

3. **OTel-Prometheus naming quirks need a project-level note** — suffix doubling (`_seconds_seconds`, `_total_total`) affects every metric assertion.

4. **Examples deserve the same review rigor as library code** — user-facing code is the first thing new adopters copy.

5. **Zero-debugging is a confirmed, reproducible process outcome** — three consecutive epics totaling 16 stories. Story clarity investment has direct, measurable ROI.

---

## Team Agreements (Effective Growth Phase)

1. **Story context accuracy** — context-creation agents must verify method signatures, feature flags, and file paths against the actual codebase, not just architecture docs.

2. **E2E queue isolation pattern** — consistency tests across interfaces must submit to queues without active workers. Document this in the E2E helper module.

3. **Example code review rigor** — examples are user-facing and deserve the same review depth as library code.

4. **Growth features require PRD + architecture update before implementation** — no coding until the design is validated.

5. **Carry forward** — pre/postcondition discipline, documentation depth mandate, security review gates, deferred-item assignment rules all remain in effect.
