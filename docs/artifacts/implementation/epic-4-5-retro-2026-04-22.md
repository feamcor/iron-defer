## Successes & Strengths

1. **Foundation compounds** — the hexagonal architecture, runtime-typed `query_as` pattern, `fresh_pool_on_shared_container()` test pattern, and figment configuration chain established in earlier epics made Epics 4 & 5 dramatically smoother. Every new capability (cancel, list, queue stats, worker status) followed the same domain → port → adapter → service → API wiring from muscle memory.

2. **100% delivery with zero scope cuts** — all 6 stories shipped, covering 16 FRs across two epics, completing the full MVP scope of 44 FRs.

3. **Code review quality improved dramatically** — Story 5.2 had just 1 review finding (1 patch). Compare to Story 3.1 which had 129 raw findings across two passes. The code quality entering review got progressively better.

4. **Story context documents compounded institutional knowledge** — every story's Previous Story Intelligence section included concrete file paths, line numbers, and signatures from prior stories. No rediscovery, no wasted motion relearning patterns.

5. **CLI refactor was clean and non-breaking** — Story 4.3 restructured from flat `CliArgs` to full subcommand architecture (`serve`, `submit`, `tasks`, `workers`, `config validate`) without breaking existing behavior. Default-to-serve via `Option<Command>` preserved backwards compatibility.

6. **Chaos tests prove the at-least-once guarantee** — 4 chaos tests with isolated Postgres containers cover worker crash recovery, DB outage survival, SIGTERM graceful shutdown, and max retries exhaustion. All pass deterministically.

7. **Supply-chain hardening completed** — `deny.toml` extended with license allow-list (13 permissive licenses) and advisory checking. `rustls-webpki` updated to fix 2 security advisories. Zero OpenSSL in the dependency tree.

## Challenges & Growth Areas

### Recurring Patterns

1. **`utoipa` research happened during implementation, not before** — the retro preparation task said "Research utoipa" but the research occurred mid-Story 4.2. The crate was added, the builder API proved too complex, and we fell back to `serde_json::json!`. Valid conclusion, but wasted motion. The OpenAPI spec is now a hand-built JSON literal — functional but not compiler-enforced against drift.

2. **Deferred work continued to accumulate** — 19 new items added from Epic 4 & 5 code reviews, bringing the total to ~42 active items. TOCTOU races, missing indexes, unbounded offsets, K8s security context, historical queue counts — all deferred with rationale but accumulating.

3. **Architecture reconciliation slipped for the second consecutive retro** — the architecture document is 18+ stories behind the implementation. Standalone cleanup tasks without a forcing function keep getting deprioritized.

4. **Newtype/typestate/bon hardening pass slipped for the second consecutive retro** — same pattern as architecture reconciliation. No `bon` crate added, no new newtypes introduced.

5. **Pre-existing clippy pedantic warnings carried through all 6 stories** — `config.rs` `duration_suboptimal_units` and `scheduler.rs` `missing_panics_doc` mentioned in every story as "not a regression."

6. **Integration test removed without replacement** — `db_outage_integration_test.rs` deleted when chaos version was created in Story 5.3, but Story 3.1 AC 7 log assertion from that test was not replaced (deferred).

## Key Lessons

1. **Foundation investment compounds** — the preparation sprint between Epics 3 and 4 paid dividends on every subsequent story. Patterns established once were reused without friction.

2. **Standalone cleanup tasks need a forcing function** — architecture reconciliation and newtype hardening slipped across two retros. They only get done when bundled into required work with a deadline.

3. **Deferred work is technical debt with a name** — 42 items accumulated across 18 stories. Each was individually reasonable to defer, but the aggregate represents significant cleanup before production.

4. **Tests and documentation are two faces of the same coin** — E2E tests validate documented workflows; tutorials grounded in tested examples stay accurate. Neither is complete without the other.

5. **Research during implementation is wasted motion** — the utoipa episode showed that preparation tasks must actually be completed before the story that depends on them.
