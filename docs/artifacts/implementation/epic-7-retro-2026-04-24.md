### Scope

- **API Query Hardening (7.1):** Offset capping at 10,000, unfiltered query warning with bounded default limit (100), pagination index `(created_at, id)`, queue statistics active-only filter via HAVING clause, case-insensitive status parsing, CLI pagination bounds clamping with saturating arithmetic, SQLx cache regeneration.
- **Payload Ownership & Dispatch Performance (7.2):** `TaskRecord.payload` migrated from `serde_json::Value` to `Arc<serde_json::Value>`, eliminating deep clones in the dispatch hot path. `payload_arc()` accessor added. `Arc::unwrap_or_clone` at HTTP boundary. Dispatch strategy benchmark (tokio::spawn vs catch_unwind) added to Criterion suite.
- **Connection Pool & Embedded Library Hardening (7.3):** `test_before_acquire` made configurable via `DatabaseConfig`. `recommended_pool_options()` public helper for embedded callers. Pool recovery interaction, acquire_timeout sizing guidance, and migration failure modes documented.
- **Kubernetes & Docker Manifests (7.4):** Startup probe (65s budget), container security context (`runAsNonRoot`, `readOnlyRootFilesystem`, `allowPrivilegeEscalation: false`), `.cargo/config.toml` copied into Docker builder stage.
- **CI Safety Gates & Deployment Validation (7.5):** GitHub Actions CI workflow with 6 quality gates (fmt, clippy, deny, test, sqlx prepare check). Readiness probe timeout (configurable, default 5s). Standalone binary fully wired (pool creation, engine build, HTTP server, worker pool, graceful shutdown). Docker Compose smoke test script.

## Challenges & Growth Areas

### Docker Container Cleanup Failure

424 zombie postgres:11-alpine containers accumulated across multiple sessions because the CLAUDE.md cleanup command filtered on `--filter "status=exited"`, but testcontainers leaves containers in **running** state. The command ran without errors but was silently ineffective.

**Root cause:** Postcondition not verified. "I ran the cleanup command" was treated as equivalent to "containers are cleaned up."

**Fix applied:** Updated CLAUDE.md cleanup command to target `ancestor=postgres:11-alpine` without status filter, catching containers regardless of state.

### K8s Validation Initially Skipped

Story 7.4 Task 4.1 (`kubectl apply --dry-run=client`) was initially marked done with "kubectl not available; YAML syntax validated via Python/manual inspection." This is not equivalent validation — Python YAML parsing doesn't check K8s API schema compliance.

**Resolution:** After minikube was installed, full server-side dry-run validation was performed and all manifests passed. The task was genuinely complete only after proper tooling was available.

## Key Lessons

1. **Pre/postcondition discipline prevents silent failures** — the Docker cleanup incident is the canonical example. Running a command is not the same as verifying its effect. Every verification task should confirm the precondition (tool availability, environment state) was met AND the postcondition (verified result) holds.

2. **Zero-debugging is reproducible, not lucky** — two consecutive epics of clean first-pass implementations confirm that story clarity investment has direct, measurable ROI. This is the project's most important process finding.

3. **Tool availability is a precondition, not an excuse** — K8s validation without kubectl is not validation. When a verification tool is unavailable, the task is blocked, not done.

4. **Type system investments compound across epics** — Epic 6's typed accessors and builder patterns directly enabled Epic 7's Arc migration. This cross-epic compounding effect is a strong argument for prioritizing type hardening early.

5. **Silent command failures are worse than loud ones** — a command that exits 0 while doing nothing useful creates false confidence. Cleanup scripts should verify their postconditions (e.g., check container count after cleanup).

---

## Team Agreements (Effective Epic 8)

1. **Pre/postcondition discipline** — before marking any verification task done, confirm the precondition (tool availability, environment state) was met AND the postcondition (verified result, not just command execution) holds. "I ran the command" is insufficient; "I verified the outcome" is required.

2. **Epic 8 documentation depth mandate** — tutorials, examples, and guides must cover the full complexity spectrum: basic to advanced, single to multi-scenario, low to high volume. Documentation is the product's adoption driver, not an afterthought. Progressive complexity: quickstart (5 min) → intermediate (priority, scheduling, retries) → advanced (multi-queue, embedded, custom errors) → production (K8s, metrics, pool tuning, graceful shutdown) → stress (10k tasks/sec, pool exhaustion, chaos).

3. **Carry forward** — security-adjacent review gates and deferred-item assignment rules from Epic 6 remain in effect.
