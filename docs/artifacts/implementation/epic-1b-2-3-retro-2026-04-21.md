## Epic Summary

### Epic 1B: Claiming & Execution Engine
Submitted tasks are automatically claimed by workers using the SKIP LOCKED atomic protocol and executed with configurable concurrency — delivering the core at-least-once guarantee.

### Epic 2: Resilience & Recovery
Tasks survive failures — automatic retry with exponential backoff, zombie recovery via sweeper, graceful shutdown that preserves in-flight work, and automatic Postgres reconnection.

### Epic 3: Observability & Compliance
Operators can monitor engine health via OTel metrics and structured logs; the task store serves as a queryable audit trail satisfying 7 regulatory frameworks.

## Key Lessons

1. **Process improvements that integrate into the existing workflow stick; standalone cleanup tasks get deprioritized.** The two fully completed retro items (#2, #4) were zero-cost additions to existing work. The three dropped items (#3, #6, #7) required dedicated time.

2. **Each story's test strategy was locally sound, but no systemic coverage assessment was done.** Per-story tests do not guarantee holistic quality. A TEA test architecture review is needed.

3. **Rust engineering standards — newtype, typestate, builder, trait ergonomics — must be codified and enforced, not aspirational.** "Parse, don't validate" at every system boundary. `bon` for builder pattern. Private fields by default.

4. **Configuration must flow through a structured chain before adding more operator-facing surfaces.** Hardcoded defaults with `#[serde(skip)]` on Duration fields will not scale to Epic 4's CLI.

5. **Technical debt is manageable now but compounds fast.** The preparation sprint is the right time to pay it down — before Epic 4 adds more integration tests, CLI commands, and REST endpoints on top.
